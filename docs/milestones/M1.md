# PaymentFlow AI — M1 Authentication & Identity Implementation Specification

Status: draft specification; not implemented.

Create the complete implementation specification for Milestone 1 (Authentication & Identity) of PaymentFlow AI.

Assume Milestone 0 (Foundation) has been fully completed and all infrastructure, repository structure, CI/CD, shared packages, developer tooling, local database, local Redis, email testing, configuration validation, observability baseline, security baseline, and health checks already exist.

The purpose of M1 is to build the complete Identity Platform that all future services depend on.

Do not redesign the architecture. Use the Product Requirements Document, System Design Document, Entity Relationship Document, OpenAPI contracts when created, Engineering Repository Blueprint, Engineering Governance and Development Standards, M0 Foundation, and this milestone specification as the source of truth.

---

## OBJECTIVE

Implement a production-ready authentication and identity platform.

By the end of M1, users should be able to:

- Register.
- Verify email.
- Login.
- Logout.
- Refresh tokens.
- Reset passwords.
- Enable MFA.
- Manage sessions.
- Manage devices.
- View profile.
- Update profile.
- Use role-based access control.
- Use permission management through approved admin flows.

The authentication platform must be enterprise-grade, secure by default, observable, auditable, and ready for future financial services.

M1 does not implement wallets, ledger posting, payments, KYC provider workflows, fraud scoring, AI Copilot behavior, or external financial integrations.

## SECTION 2 — Authentication Architecture

M1 implements an identity platform composed of focused services and layers.

### Identity Service

Responsibilities:

- Own user identity records.
- Own profile records.
- Own roles and permissions.
- Provide user lookup APIs for internal services.
- Provide profile read/update APIs.
- Publish identity lifecycle events.

### Authentication Service

Responsibilities:

- Register users.
- Authenticate credentials.
- Verify email.
- Manage password reset.
- Issue and refresh tokens.
- Create and revoke sessions.
- Enforce account lockout.
- Coordinate MFA challenges.

### Authorization Layer

Responsibilities:

- Enforce RBAC and permissions.
- Provide authorization middleware for API Gateway and services.
- Validate authenticated identity context.
- Support future tenant-aware authorization checks.
- Return safe 401 and 403 errors.

### Token Service

Responsibilities:

- Generate access tokens.
- Generate refresh tokens.
- Hash and store refresh tokens.
- Rotate refresh tokens.
- Revoke refresh tokens.
- Detect refresh-token reuse.
- Manage signing keys and key rotation strategy.

### Session Service

Responsibilities:

- Create sessions during login.
- Track active sessions.
- List sessions.
- Revoke a single session.
- Revoke all sessions.
- Store metadata for IP, user agent, device, and last seen time.

### Device Service

Responsibilities:

- Register devices.
- Track trusted devices.
- Store device metadata.
- Revoke devices.
- Detect suspicious or new devices.
- Support device-aware MFA and security alerts.

### Email Verification Service

Responsibilities:

- Generate verification tokens.
- Store token hashes and expirations.
- Verify email addresses.
- Resend verification emails.
- Audit verification actions.

### Password Reset Service

Responsibilities:

- Generate password reset tokens.
- Store token hashes and expirations.
- Validate reset requests.
- Enforce password policy and history.
- Revoke sessions after password reset.
- Emit security alerts.

### Interactions

```text
Client -> API Gateway -> Auth Service -> Identity DB
Auth Service -> Redis (rate limits, temporary challenge state)
Auth Service -> Email Service -> Mailpit/provider
Auth Service -> Audit Events
API Gateway -> Authorization Middleware -> Identity/Auth lookup
Admin App -> API Gateway -> Admin Identity APIs -> Identity Service
```

### Architectural Rules

- Auth Service owns authentication behavior.
- Identity Service owns user, profile, role, and permission data.
- No future service should parse credentials or manage sessions directly.
- Future services must rely on signed tokens, introspection, or authorization middleware.
- Identity APIs must not expose secrets, token hashes, password hashes, MFA secrets, or internal risk signals.

## SECTION 3 — Database Implementation

M1 must implement all Identity tables and migrations.

### Required Schema

Use the `identity` schema for M1 tables.

### Required Entities

- `users`
- `roles`
- `permissions`
- `user_roles`
- `sessions`
- `refresh_tokens`
- `devices`
- `mfa_methods`
- `email_verifications`
- `password_resets`
- `login_attempts`
- `audit_events`

Additional supporting tables are allowed when needed, including `user_profiles`, `user_preferences`, `password_history`, `backup_codes`, and `role_permissions`.

### `identity.users`

Columns:

- `id` UUID primary key.
- `email` case-insensitive unique nullable text.
- `username` case-insensitive unique nullable text.
- `phone` unique nullable text.
- `status` enum: `pending_verification`, `active`, `locked`, `suspended`, `closed`.
- `email_verified_at` nullable timestamp.
- `phone_verified_at` nullable timestamp.
- `created_at` timestamp.
- `updated_at` timestamp.

Indexes:

- Unique index on normalized email.
- Unique index on normalized username where username is supported.
- Unique index on phone where present.
- Index on status.

Constraints:

- At least one login identifier must be present.
- Email must be normalized before storage.
- Status must use a controlled enum.

### `identity.roles`

Columns:

- `id` UUID primary key.
- `name` unique text.
- `description` text.
- `is_system_role` boolean.
- `created_at` timestamp.
- `updated_at` timestamp.

Seed roles:

- Consumer.
- Freelancer.
- Business Owner.
- Employee.
- Finance Manager.
- Admin.
- Super Admin.

### `identity.permissions`

Columns:

- `id` UUID primary key.
- `name` unique text.

- `created_at` timestamp.

Required permission groups:

- Roles.
- Permissions.

## SECTION 4 — User Registration

### Registration Endpoint

Implement:

- `POST /auth/register`

### Request Requirements

- Email.
- Username if supported.
- Password.
- Optional phone.
- Accepted terms flag where required.
- Locale/timezone defaults where provided.

### Validation

- Email format.
- Username format if supported.
- Password policy.
- Duplicate email detection.
- Duplicate username detection where supported.
- Phone format where provided.
- Required consent flags.

### Duplicate Detection

- Duplicate responses must be safe and must not leak unnecessary account details.
- Duplicate registration attempts should not create multiple user records.

### Email Verification Token Generation

- Generate cryptographically secure token.
- Store only token hash.
- Set expiration.
- Track resend count.

### Password Hashing

- Use Argon2.
- Never log plaintext passwords.
- Never store plaintext passwords.
- Password hashing parameters must be documented and configurable by environment where safe.

### Audit Logging

Create audit events for:

- Registration requested.
- User created.
- Verification email sent.
- Registration rejected.

### Welcome Notification

- Send welcome notification only after email verification or after the approved activation point.
- Use local Mailpit during development and tests.

### Acceptance Criteria

- Valid registration creates a pending user.
- Duplicate email is handled safely.
- Invalid password is rejected.
- Verification email is generated.
- Audit events are created.
- No plaintext secret appears in logs or database.

### Business Rules

- New users begin in `pending_verification` unless a verified identity method already exists.
- Users cannot access protected financial features before later milestones authorize them.
- Registration is rate-limited.

### Error Handling

- Validation errors use standard problem-details shape.
- Rate-limit errors are explicit and safe.
- Duplicate errors are safe.
- Email provider failure is observable and retryable where appropriate.

## SECTION 5 — Email Verification

### Verification Token Generation

- Tokens must be cryptographically random.
- Only token hashes are stored.
- Tokens must be single-use.

### Expiration

- Verification tokens must expire.
- Expired tokens must not verify an email.
- Expired-token responses must allow safe resend.

### Verification Endpoint

Implement:

- `POST /auth/verify-email`

Workflow:

1. Validate token.
2. Hash token and lookup active verification record.
3. Confirm not expired and not used.
4. Mark email as verified.
5. Activate user where business rules allow.
6. Mark token used.
7. Emit audit event.

### Resend Verification

Implement:

- `POST /auth/resend-verification`

Rules:

- Enforce resend rate limits.
- Invalidate or expire older tokens according to policy.
- Do not reveal whether an arbitrary email exists.

### Failure Handling

- Invalid token returns safe error.
- Expired token returns safe error and resend option.
- Already verified email returns idempotent safe response.
- Too many resends triggers rate limit.

- Email login.
- Username login if username is enabled.

### Password Validation

- Validate identifier and password.
- Use Argon2 verification.
- Use constant-time comparison where applicable.
- Record login attempts.

### Refresh Tokens

- Generate refresh token during successful login.
- Store only token hash.
- Bind refresh token to session.
- Rotate refresh tokens on refresh.

### JWT Generation

Access tokens must include:

- Issuer.
- Subject.
- Audience.
- Issued-at.
- Expiration.
- Token ID.
- Roles or permission scopes.
- Session ID.

### Session Creation

- Create session after successful login.
- Store IP, user-agent hash, device, timestamps, and expiration.

### Device Registration

- Detect new device.
- Create or update device record.
- Trigger MFA or security alert for suspicious devices.

### Audit Logging

Audit:

- Login success.
- Login failure.
- MFA required.
- New device detected.
- Suspicious login blocked.

### Rate Limiting

- Rate-limit by identifier.
- Rate-limit by IP.
- Apply account lockout after repeated failures.

### Failure Rules

- Invalid credentials must not reveal whether email or username exists.
- Locked or suspended account responses must be safe and actionable.
- Unverified users may be blocked or limited according to policy.

## SECTION 7 — Logout

### Single Device Logout

Implement:

- `POST /auth/logout`

Behavior:

- Revoke current refresh token.
- Revoke current session.
- Emit audit event.
- Return idempotent success.

### Logout All Devices

Implement:

- `DELETE /sessions`

Behavior:

- Revoke all active sessions for the user.
- Revoke all active refresh tokens.
- Keep current action auditable.

### Refresh Token Revocation

- Token revocation must be persisted.
- Reused revoked tokens must be detected.
- Reuse detection should revoke related session family.

### Session Termination

- Terminated sessions must no longer refresh tokens.
- Active access tokens expire naturally unless token denylisting is introduced for high-risk revocation.

## SECTION 8 — Password Management

### Forgot Password

Implement:

- `POST /auth/forgot-password`

Rules:

- Always return safe response.
- Generate reset token for existing users.
- Store only token hash.
- Rate-limit requests.
- Send reset email.

### Reset Password

Implement:

- `POST /auth/reset-password`

Workflow:

1. Validate reset token.
2. Enforce password policy.
3. Check password history.
4. Hash new password with Argon2.
5. Store password.
6. Mark reset token used.
7. Revoke existing sessions unless policy allows current session.
8. Send security alert.
9. Emit audit event.

### Password Policy

Minimum requirements:

- Length threshold.
- Common-password rejection.
- Identifier/email similarity rejection.
- Password history check.
- Optional complexity rules if approved.

### Password History

- Store password hashes in history.
- Reject reuse according to configured history length.

### Password Expiration

- Optional.
- Required only for admin roles if policy demands it.
- Must not create unsafe forced-reset loops.

### Security Validation

- No plaintext password logs.
- No plaintext password audit metadata.
- Reset tokens are single-use and expiring.
- Password reset cannot enumerate accounts.

## SECTION 12 — Role-Based Access Control

### Roles

Default roles:

- Consumer.
- Freelancer.
- Business Owner.
- Employee.
- Finance Manager.
- Admin.
- Super Admin.

### Permissions

Permissions must be explicit and resource/action based.

Examples:

- `users:read:self`
- `users:update:self`
- `sessions:read:self`
- `sessions:revoke:self`
- `roles:assign`
- `permissions:manage`
- `admin:users:search`
- `admin:users:suspend`
- `audit:read`

### Permission Inheritance

- Super Admin inherits all admin permissions.
- Admin receives operational permissions but not unrestricted system control unless explicitly granted.
- Finance Manager permissions should prepare for future financial workflows but must not authorize ledger posting in M1.

### Authorization Middleware

Middleware must:

- Validate access token.
- Attach identity context.
- Enforce required roles and permissions.
- Deny cross-user access by default.
- Return safe 401/403 responses.

### Policy Engine

M1 must define a policy evaluation layer that can later support:

- Role checks.
- Permission checks.
- Resource ownership checks.
- Tenant/company checks.
- MFA step-up checks.
- Future verification/KYC status checks.

### Admin Role Safety

- Admin and Super Admin actions require MFA.
- Role assignment is audited.
- Super Admin bootstrap must be controlled through seed or secure setup flow.

## SECTION 13 — Security

### Argon2 Password Hashing

- Use Argon2 for password hashing.
- Document memory, time, and parallelism parameters.
- Allow safe parameter upgrades.

### JWT

- Use signed access tokens.
- Use short expirations.
- Include minimal claims.
- Support signing key rotation strategy.

### Refresh Tokens

- Use opaque refresh tokens.
- Store only token hashes.
- Bind tokens to sessions.
- Rotate on every refresh.

### Token Rotation

- Refresh token reuse must be detected.
- Reuse detection must revoke affected token family/session.

### Account Lockout

- Lock or challenge accounts after repeated failed attempts.
- Lockout must be time-bound or admin-resolvable.
- Lockout must be audited.

### Rate Limiting

- Rate-limit registration.
- Rate-limit login.
- Rate-limit password reset.
- Rate-limit email verification resend.
- Rate-limit MFA verification.

### IP Monitoring

- Store IP metadata for security analysis.
- Flag unusual login patterns.
- Avoid storing excessive location detail.

### Brute-Force Protection

- Use login attempt tracking.
- Apply exponential backoff or lockout policy.
- Return safe errors.

### Security Headers

- API Gateway and frontend must use baseline security headers from M0.

### Session Expiration

- Access tokens expire quickly.
- Refresh tokens expire according to risk policy.
- Sessions expire and can be revoked.

### CSRF Strategy

- If browser cookies are used, implement CSRF protection.
- If bearer tokens are used in Authorization headers, document CSRF posture and XSS protections.

### Secrets

- No secrets in logs.
- No secrets in audit metadata.
- Secrets loaded from validated config.
