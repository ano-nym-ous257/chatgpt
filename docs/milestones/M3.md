# PaymentFlow AI — M3 Wallet Service Implementation Specification

Status: draft specification; not implemented.

Create the complete implementation specification for Milestone 3 (Wallet Service) of PaymentFlow AI.

Assume the following milestones have already been completed:

- M0 Foundation.
- M1 Authentication & Identity.
- M2 Ledger Engine.

The Wallet Service must consume the Ledger Engine and never bypass it. The Ledger remains the immutable source of truth. Wallet balances are read-optimized projections derived from ledger postings.

Do not redesign the architecture. Use the Product Requirements Document, System Design Document, Entity Relationship Document, OpenAPI contracts when created, Engineering Repository Blueprint, Engineering Governance and Development Standards, Enterprise Engineering Implementation Plan, previous milestone specifications, and this milestone specification as the source of truth.

---

## OBJECTIVE

Implement a production-ready Wallet Service supporting:

- Personal wallets.
- Business wallets.
- Multi-currency wallets.
- Wallet accounts.
- Balance projection.
- Available balance.
- Pending balance.
- Reserved balance.
- Wallet limits.
- Wallet status.
- Currency conversion readiness.
- Wallet APIs.
- Wallet analytics.
- Wallet security.

The Wallet Service should provide a fast, scalable user-facing interface while relying on the Ledger Engine for financial integrity.

M3 does not implement payment orchestration, external payment provider integrations, payroll execution, invoice settlement, KYC/KYB provider workflows, fraud scoring, or treasury automation. It may expose the wallet primitives those future services require.

### Objectives

- Implement configurable limits and velocity checks.
- Implement reservation workflows for pending financial operations.
- Publish wallet events for analytics, notifications, and future payments.
- Provide secure APIs for wallet creation, listing, lookup, balance reads, reservations, status changes, and transaction summaries.

### Deliverables

- Wallet Service implementation.
- Wallet database migrations.
- Wallet account manager.
- Balance projection engine.
- Currency manager.
- Limit engine.
- Reservation engine.
- Wallet lifecycle manager.
- Wallet event publisher.
- Wallet API contracts.
- Wallet analytics metrics.
- Unit, integration, contract, load, concurrency, projection, reservation, and currency tests.
- Observability dashboards, metrics, traces, logs, and alerts.
- Handoff documentation for M4 Payments Service.

### Dependencies

- M0 Foundation completed and accepted.
- M1 Authentication & Identity completed and accepted.
- M2 Ledger Engine completed and accepted.
- Authenticated user and service identity available.
- RBAC and authorization middleware available.
- Ledger APIs available for account creation, posting, balance reads, reversals, and audit references.
- Event bus and shared event envelope available.
- Local and CI database migration tooling available.

### Business Value

Wallets are the primary product surface for users and businesses. They translate ledger-backed financial state into fast, understandable account experiences while preserving the ledger as the only financial source of truth. The Wallet Service enables balance views, wallet lifecycle management, limit checks, reservations, payment readiness, analytics, and future treasury workflows.

### Risks

- Treating wallet balances as source-of-truth balances.
- Updating projected balances directly instead of consuming ledger postings.
- Race conditions during reservations and concurrent balance reads.
- Inconsistent currency precision and rounding.
- Incorrect lifecycle transitions that allow frozen or closed wallets to transact.
- Limits that are checked in application code but bypassed by future services.
- Cache drift between wallet projections and ledger balances.
- Unauthorized wallet access across users, businesses, or tenants.
- Reservation leakage where expired holds reduce available balance.

### Definition of Done

M3 is complete only when wallet creation, wallet account creation, ledger account mapping, balance projection, reservation lifecycle, limit enforcement, lifecycle transitions, multi-currency support, wallet APIs, wallet events, analytics, security controls, tests, observability, and M4 handoff are complete and validated by the acceptance tests in Section 18.

### Acceptance Criteria

- Users can create wallets where allowed.
- Businesses can own wallets where allowed.
- Wallet accounts map to ledger accounts.
- Balances are projected from ledger postings.
- No endpoint directly mutates balances outside the ledger-backed projection workflow.
- Multi-currency wallet behavior works for supported currencies.
- Reservations affect available balance through ledger-safe reservation workflows.
- Wallet lifecycle rules are enforced.
- Limits are enforced.
- Wallet APIs meet performance targets.
- Events are published for wallet lifecycle and projection changes.
- All tests pass.
- CI passes.

## SECTION 2 — Wallet Architecture

M3 implements a Wallet Service that provides wallet-domain APIs and read models while delegating all financial truth to the Ledger Service.

### Wallet Service

Responsibilities:

- Own wallet metadata.
- Own wallet lifecycle states.
- Own wallet settings.
- Own wallet limits.
- Own wallet reservations as operational intents.
- Own wallet event publishing.
- Serve wallet and balance APIs.
- Maintain wallet read projections from ledger events.
- Enforce wallet authorization and lifecycle rules.

Non-responsibilities:

- It must not create unbalanced financial postings.
- It must not mutate ledger journal entries directly.
- It must not bypass Ledger Service APIs.
- It must not treat wallet projection rows as financial source of truth.
- It must not orchestrate external provider payments.

### Balance Projection Engine

Responsibilities:

- Consume committed ledger events.
- Rebuild wallet balance projections from ledger account balances and journal lines.
- Maintain current, available, pending, reserved, locked, and historical balance views.
- Detect projection drift.
- Reconcile projections against Ledger Service balances.
- Publish `wallet.balance_updated` events after projection changes.

Rules:

- Projection updates are derived from ledger commits.
- Projection rows may be cached for reads.
- Projection drift must be alertable.
- Projection rebuild must be idempotent.

### Wallet Account Manager

Responsibilities:

- Create wallet accounts per wallet and currency.
- Request corresponding ledger accounts from Ledger Service.
- Maintain mapping between wallet accounts and ledger accounts.
- Support account purposes such as available, pending, reserved, locked, settlement, escrow, and fee where required.
- Enforce one active wallet-account mapping per wallet, currency, and purpose.

### Currency Manager

Responsibilities:

- Maintain supported currency definitions.
- Enforce currency precision.
- Validate wallet currency restrictions.
- Store preferred currency.
- Provide display formatting rules.
- Cache exchange-rate data for display and quote-readiness.

Rules:

- Currency conversion financial postings belong to Ledger Service and future Payments/FX flows.
- M3 may expose conversion readiness and quote display support only where backed by M2 ledger/FX primitives.

### Limit Engine

Responsibilities:

- Validate daily, monthly, single transaction, maximum balance, minimum balance, and velocity limits.
- Enforce default limits by wallet type, user role, tenant, and currency.
- Expose limit evaluation results for future Payments Service.
- Store limit overrides with auditability.

### Reservation Engine

Responsibilities:

- Create reservation intent records.
- Request ledger reservation postings where funds must be held.
- Release reservations.
- Expire reservations.
- Ensure available balance reflects active ledger-backed holds.
- Prevent duplicate reservations through idempotency.

Rules:

- Financial reservations must be represented by ledger postings or ledger-recognized reserved accounts.
- `wallet_reservations` records track operational lifecycle and references; they are not the financial source of truth.

### Wallet Event Publisher

Responsibilities:

- Publish lifecycle events.
- Publish balance projection events.
- Publish reservation events.
- Publish currency preference events.
- Use the shared event envelope from M0/M2.
- Include correlation and causation IDs.

### Interaction with Ledger Service

The Wallet Service uses Ledger Service to:

- Create ledger accounts.
- Retrieve ledger account balances.
- Subscribe to committed ledger journal events.
- Request reservation postings.
- Request release postings.
- Validate ledger account mappings.
- Rebuild balance projections.

The Wallet Service must never write ledger tables directly.

### Interaction with Identity Service

The Wallet Service uses Identity Service to:

- Validate authenticated users.
- Evaluate roles and permissions.
- Resolve user profile and status.
- Enforce user-level wallet ownership.
- Apply verification or restriction status when available.

### Interaction with Notification Service

The Wallet Service may trigger notifications for:

- Wallet created.
- Wallet activated.
- Wallet frozen.
- Wallet unfrozen.
- Wallet closed.
- Reservation created.
- Reservation released.
- Balance threshold reached.

Notification delivery is owned by Notification Service.

### Interaction with Analytics Service

The Wallet Service publishes events and metrics consumed by analytics for:

- Active wallet growth.
- Currency distribution.
- Balance trends.
- Dormancy analysis.
- Reservation behavior.
- Velocity statistics.

## SECTION 3 — Database Implementation

M3 owns the `wallets` schema. During the MVP, cross-schema foreign keys may exist where all schemas share one managed PostgreSQL deployment. At scale, cross-domain references must become logical UUID references validated by service APIs and events.

### Required Entities

- `wallets`
- `wallet_accounts`
- `wallet_balances`
- `wallet_limits`
- `wallet_reservations`
- `wallet_status_history`
- `wallet_settings`
- `wallet_events`
- `currency_definitions`
- `exchange_rate_cache`

### `wallets.wallets`

Purpose:

- Stores wallet metadata and lifecycle state.

Columns:

- `id` UUID primary key.
- `owner_type` enum: `consumer`, `business`, `treasury`, `settlement`, `escrow`, `system`.
- `owner_id` UUID nullable for system wallets.
- `wallet_type` enum: `consumer`, `business`, `treasury`, `settlement`, `escrow`, `system`.
- `display_name` text.
- `status` enum: `created`, `active`, `restricted`, `frozen`, `suspended`, `closed`, `archived`.
- `default_currency` CHAR(3).
- `preferred_currency` CHAR(3).
- `region` text nullable.
- `created_by` UUID nullable.
- `created_at` timestamp.
- `updated_at` timestamp.
- `activated_at` nullable timestamp.
- `closed_at` nullable timestamp.
- `archived_at` nullable timestamp.

Indexes:

- `idx_wallets_owner` on `(owner_type, owner_id)`.
- `idx_wallets_status` on `status`.
- `idx_wallets_type` on `wallet_type`.
- `idx_wallets_default_currency` on `default_currency`.
- `idx_wallets_created_at` on `created_at`.

Constraints:

- `default_currency` references `wallets.currency_definitions(code)`.
- `preferred_currency` references `wallets.currency_definitions(code)`.
- `owner_id` is required unless `owner_type='system'`.
- `closed_at` must be set when status is `closed`.
- `archived_at` must be set when status is `archived`.

Relationships:

- One wallet has many wallet accounts.
- One wallet has many limits.
- One wallet has many reservations.
- One wallet has many status history rows.
- One wallet has one settings row.

Foreign Keys:

- `default_currency` → `currency_definitions.code`.
- `preferred_currency` → `currency_definitions.code`.
- `owner_id` is a logical reference to Identity or Business domain owner.

Partitioning Strategy:

- No partitioning required initially.
- At scale, partition by `region` or hash of `owner_id` if wallet volume requires it.

Retention Policy:

- Wallet metadata is retained for the regulatory retention period after closure.
- Closed wallets may be archived but not physically deleted while audit retention applies.

### `wallets.wallet_accounts`

Purpose:

- Maps wallet currency/purpose accounts to Ledger Service accounts.

Columns:

- `id` UUID primary key.
- `wallet_id` UUID not null.
- `currency` CHAR(3) not null.
- `account_purpose` enum: `available`, `pending`, `reserved`, `locked`, `settlement`, `escrow`, `fee`.
- `ledger_account_id` UUID not null.
- `account_code` text unique not null.
- `status` enum: `active`, `restricted`, `closed`.
- `created_at` timestamp.
- `updated_at` timestamp.
- `closed_at` nullable timestamp.

Indexes:

- `idx_wallet_accounts_wallet` on `wallet_id`.
- `idx_wallet_accounts_currency` on `currency`.
- `idx_wallet_accounts_ledger_account` on `ledger_account_id`.
- `idx_wallet_accounts_status` on `status`.

Constraints:

- `wallet_id` references `wallets.wallets(id)`.
- `currency` references `wallets.currency_definitions(code)`.
- `ledger_account_id` must map to an active ledger account.
- Unique `(wallet_id, currency, account_purpose)` while active.
- Closed accounts cannot be reused.

Relationships:

- Many wallet accounts belong to one wallet.
- One wallet account has one projected balance row.
- One wallet account maps to one ledger account.

Foreign Keys:

- `wallet_id` → `wallets.wallets(id)`.
- `currency` → `wallets.currency_definitions(code)`.
- `ledger_account_id` is a logical or physical reference to `ledger.ledger_accounts(id)`.

Partitioning Strategy:

- No partitioning initially.
- At scale, hash partition by `wallet_id` or `ledger_account_id`.

Retention Policy:

- Retain wallet-account mappings for the same period as wallet metadata and ledger audit references.

### `wallets.wallet_balances`

Purpose:

- Stores read-optimized balance projections derived from ledger postings.

Columns:

- `id` UUID primary key.
- `wallet_account_id` UUID not null.
- `wallet_id` UUID not null.
- `currency` CHAR(3) not null.
- `current_balance` NUMERIC(30,8) not null default 0.
- `available_balance` NUMERIC(30,8) not null default 0.
- `pending_balance` NUMERIC(30,8) not null default 0.
- `reserved_balance` NUMERIC(30,8) not null default 0.
- `locked_balance` NUMERIC(30,8) not null default 0.
- `ledger_balance_version` BIGINT nullable.
- `last_ledger_journal_id` UUID nullable.
- `projection_version` BIGINT not null default 0.
- `as_of` timestamp not null.
- `created_at` timestamp.
- `updated_at` timestamp.

Indexes:

- Unique index on `wallet_account_id`.
- `idx_wallet_balances_wallet` on `wallet_id`.
- `idx_wallet_balances_currency` on `currency`.
- `idx_wallet_balances_as_of` on `as_of`.
- `idx_wallet_balances_available` on `available_balance`.

Constraints:

- `wallet_account_id` references `wallet_accounts(id)`.
- `wallet_id` references `wallets(id)`.
- `currency` references `currency_definitions(code)`.
- Balances must respect configured currency precision.
- Negative balances are prohibited unless the wallet account policy explicitly allows overdraft or negative accounting classification.

Relationships:

- One wallet account has one active projection row.
- Historical balance snapshots may be produced through events or future analytics tables.

Foreign Keys:

- `wallet_account_id` → `wallets.wallet_accounts(id)`.
- `wallet_id` → `wallets.wallets(id)`.
- `currency` → `wallets.currency_definitions(code)`.
- `last_ledger_journal_id` is a logical reference to Ledger Service.

Partitioning Strategy:

- No partitioning initially.
- At scale, hash partition by `wallet_id`.

Retention Policy:

- Current projection rows are retained while wallet account exists.
- Historical projections should be retained through `wallet_events` or analytics warehouse snapshots.

### `wallets.wallet_limits`

Purpose:

- Stores configurable wallet limits and velocity controls.

Columns:

- `id` UUID primary key.
- `wallet_id` UUID not null.
- `currency` CHAR(3) nullable.
- `limit_type` enum: `daily_transaction`, `monthly_transaction`, `single_transaction`, `maximum_balance`, `minimum_balance`, `velocity_count`, `velocity_amount`.
- `limit_value` NUMERIC(30,8) not null.
- `window_seconds` integer nullable.
- `scope` enum: `wallet`, `wallet_account`, `owner`, `tenant`, `currency`.
- `effective_from` timestamp.
- `effective_until` nullable timestamp.
- `created_by` UUID nullable.
- `reason` text nullable.
- `created_at` timestamp.
- `updated_at` timestamp.

Indexes:

- `idx_wallet_limits_wallet` on `wallet_id`.
- `idx_wallet_limits_type` on `limit_type`.
- `idx_wallet_limits_currency` on `currency`.
- `idx_wallet_limits_effective` on `(effective_from, effective_until)`.

Constraints:

- `wallet_id` references `wallets(id)`.
- `currency` references `currency_definitions(code)` when present.
- `limit_value` must be greater than or equal to zero.

Relationships:

- One wallet may have many limits.
- Limits may override default policy by wallet, owner, tenant, or currency.

Foreign Keys:

- `wallet_id` → `wallets.wallets(id)`.
- `currency` → `wallets.currency_definitions(code)`.
- `created_by` is a logical reference to Identity Service user ID.

Partitioning Strategy:

- No partitioning required initially.

Retention Policy:

- Expired limits must be retained for audit.
- Limit overrides are retained according to audit retention rules.

### `wallets.wallet_reservations`

Purpose:

Columns:

- `wallet_id` UUID not null.
- `wallet_account_id` UUID not null.
- `currency` CHAR(3) not null.
- `amount` NUMERIC(30,8) not null.
- `reservation_type` enum: `pending_payment`, `invoice_payment`, `payroll_processing`, `escrow`, `compliance_hold`.

## SECTION 4 — Wallet Types

M3 must support multiple wallet types with explicit lifecycle rules.

### Consumer Wallet

Purpose:

- Personal user wallet for retail users.

Rules:

- Owner must be an authenticated user.
- One user may have one or more consumer wallets depending on product policy.
- Wallet may support multiple currencies.
- Activation may depend on verification status from Identity/Compliance placeholders.
- Consumer wallets cannot use treasury-only operations.

Lifecycle:

- Created after request validation.
- Active after eligibility checks.
- Restricted, frozen, or suspended when risk/compliance/admin rules require.
- Closed only when balances are zero or swept according to ledger-safe process.

### Business Wallet

Purpose:

- Wallet owned by a business or company entity.

Rules:

- Owner must be a business/company identifier or approved placeholder.
- Authorized business roles may create or manage wallet settings.
- Finance Manager or Business Owner role may view balances according to permissions.
- Multi-user access must be role-controlled.

Lifecycle:

- Created by authorized business user.
- Active after business eligibility requirements.
- Restricted/frozen/suspended through admin or compliance controls.
- Closed only after all currency accounts are zeroed or swept.

### Treasury Wallet

Purpose:

- Internal or business treasury wallet for liquidity and cash-management workflows.

Rules:

- Creation requires elevated permissions.
- Used by future Treasury Service.
- May have stricter audit and approval rules.
- Must not be available to normal consumer flows.

Lifecycle:

- Created by authorized admin or future treasury workflow.
- Active only after approval.
- Frozen or restricted during treasury risk events.
- Closed through controlled operational process.

### Settlement Wallet

Purpose:

- Wallet representing settlement balances for internal or provider-facing workflows.

Rules:

- Usually system-owned.
- Creation requires system/admin authority.
- Used by future Payments/Provider flows.
- Must map to ledger settlement accounts.

Lifecycle:

- Created through operational setup.
- Active while settlement workflow is active.
- Frozen during reconciliation or provider incidents.
- Closed only after reconciliation and zero-balance confirmation.

### Escrow Wallet

Purpose:

- Wallet used to hold funds pending condition fulfillment.

Rules:

- Must use ledger-backed reserved/escrow accounts.
- Release requires explicit reference and authorization.
- Expiration or dispute rules must be documented by future consuming services.

Lifecycle:

- Created for approved escrow use case.
- Active while conditions are pending.
- Restricted during dispute or compliance review.
- Closed after funds are released, reversed, or resolved.

### System Wallet

Purpose:

- Internal wallet used for platform operational accounts such as fees, suspense, reserves, and internal settlement.

Rules:

- System-owned.
- Not visible to normal users.
- Access requires service or admin permissions.
- Every system wallet must have clear ledger account mapping and purpose.

Lifecycle:

- Created through controlled setup.
- Active while platform account purpose exists.
- Frozen during operational incident.
- Closed only through audited process.

## SECTION 5 — Balance Model

Balances are projections from ledger entries. The Wallet Service must never directly manipulate balances as financial truth.

### Current Balance

Definition:

- The ledger-derived balance for a wallet account at a point in time.

Rules:

- Derived from committed ledger postings.
- May be read from wallet projection cache.
- Must reconcile with Ledger Service balance.

### Available Balance

Definition:

- Amount available for new operations after reserved, pending, locked, and limit effects are considered.

Formula:

```text
available_balance = current_balance - reserved_balance - locked_balance - pending_debit_effects
```

Rules:

- Must not include expired reservations.
- Must not include uncommitted ledger changes as final truth.
- If a reservation is pending ledger confirmation, the API must show the reservation state clearly.

### Pending Balance

Definition:

- Amount associated with unsettled or pending ledger-backed operations.

Rules:

- Derived from ledger pending accounts or committed pending-state postings.
- Used for payments, incoming funds, and future settlement flows.

### Reserved Balance

Definition:

- Amount held for a specific purpose such as pending payment, invoice payment, payroll run, escrow, or compliance hold.

Rules:

- Financial holds require ledger-backed reservation postings.
- Active reservation records must reference ledger reservation journals where applicable.
- Release requires a ledger release posting or approved ledger reversal flow.

### Locked Balance

Definition:

- Amount unavailable because of wallet freeze, compliance hold, account lock, or administrative restriction.

Rules:

- Locked balance must be explainable by status, reservation, or ledger account purpose.
- Locked amounts cannot be spent by future Payments Service.

### Historical Balance

Definition:

- Balance at a previous point in time.

Rules:

- Derived from ledger history, ledger snapshots, or wallet projection snapshots.
- Wallet historical views must state the `as_of` timestamp.
- Ledger remains authoritative for audit disputes.

### Balance Calculation Workflow

1. Ledger Service commits journal entries.
2. Ledger Service emits committed ledger event.
3. Wallet projection consumer receives event.
4. Wallet Service maps ledger account IDs to wallet accounts.
5. Wallet Service recalculates affected projection rows.
6. Projection update stores ledger version and last journal reference.
7. Wallet Service publishes balance-updated event.
8. Read API returns projection with `as_of`, `projection_version`, and currency precision.

### Projection Rebuild Workflow

1. Select wallet or wallet account.
2. Retrieve authoritative ledger account balances and/or journal history.
3. Recompute current, available, pending, reserved, and locked balances.
4. Compare recomputed values to current projection.
5. Update projection if different.
6. Emit drift metric and audit/log entry if drift exceeded tolerance.

### Direct Manipulation Prohibition

No API, admin action, migration, cron job, test helper, or script may change wallet balances as financial truth. Adjustments must be ledger postings from M2 rules.

## SECTION 6 — Wallet Lifecycle

Wallet lifecycle states must be explicit and enforced everywhere.

### States

- `created`
- `active`
- `restricted`
- `frozen`
- `suspended`
- `closed`
- `archived`

### Created

Meaning:

- Wallet metadata exists, but full use may not yet be allowed.

Allowed:

- Complete setup.
- Create required wallet accounts.
- Activate when eligibility passes.

Blocked:

- Payments.
- Withdrawals.
- Financial reservations unless explicitly approved for setup.

### Active

Meaning:

- Wallet is usable for supported operations.

Allowed:

- Balance reads.
- Reservations.
- Future payment initiation.
- Currency account creation where allowed.
- Settings updates.

Blocked:

- Operations outside wallet type, role, currency, or limit policy.

### Restricted

Meaning:

- Wallet has partial limitations.

Allowed:

- Balance reads.
- Incoming funds if policy allows.
- Release of reservations if policy allows.
- Compliance/admin remediation flows.

Blocked:

- New outgoing reservations unless explicitly allowed.
- Currency expansion unless approved.

### Frozen

Meaning:

- Wallet is temporarily locked for risk, compliance, administrative, or user-requested reasons.

Allowed:

- Balance reads.
- Admin review.
- Audit and reporting.

Blocked:

- New financial operations.
- New reservations.
- Settings that affect financial behavior.

### Suspended

Meaning:

- Wallet is disabled due to serious policy, risk, or account status reason.

Allowed:

- Admin review.
- Balance reads for authorized actors.
- Required regulatory reporting.

Blocked:

- User-initiated financial operations.
- New wallet accounts.
- Currency changes.

### Closed

Meaning:

- Wallet is no longer usable, but retained for audit.

Allowed:

- Audit reads.
- Statement/history reads if policy allows.

Blocked:

- New wallet accounts.
- Reservations.
- Payments.
- Settings changes except archival metadata.

### Archived

Meaning:

- Wallet is retained in low-activity or historical storage state.

Allowed:

- Audit retrieval.
- Compliance retrieval.

Blocked:

- All user-facing operations except historical access where policy permits.

### Transition Rules

- `created` → `active` after setup and eligibility checks.
- `active` → `restricted` when partial limitations apply.
- `restricted` → `active` after restrictions are cleared.
- `active` or `restricted` → `frozen` for temporary lock.
- `frozen` → `active` or `restricted` after review.
- Any non-closed state → `suspended` for serious policy violations.
- `active`, `restricted`, `frozen`, or `suspended` → `closed` only after ledger-safe closure conditions.
- `closed` → `archived` after retention and archival rules allow.

### Business Constraints

- Closed wallets must have no active reservations.
- Closed wallets must have zero or ledger-swept balances according to policy.
- Frozen wallets cannot create new reservations.
- Suspended wallets cannot be reactivated without authorized admin action.
- Every status change must write status history and publish an event.

## SECTION 9 — Reservations

Reservations allow future operations to hold funds safely before final execution.

### Reservation Use Cases

- Pending payments.
- Invoice payments.
- Payroll processing.
- Escrow.
- Compliance holds.

### Reservation Lifecycle

States:

- `pending_post`
- `active`
- `released`
- `expired`
- `cancelled`
- `failed`

### Create Reservation Workflow

1. Authenticate and authorize actor.
2. Validate wallet status.
3. Validate wallet account and currency.
4. Validate amount and precision.
5. Check idempotency key.
6. Check limits.
7. Check available balance projection.
8. Create reservation intent with `pending_post`.
9. Request ledger reservation posting from Ledger Service.
10. On ledger commit, mark reservation `active`.
11. Update projection from ledger event.
12. Publish `wallet.reservation_created`.

### Release Reservation Workflow

1. Authenticate and authorize actor.
2. Load active reservation.
3. Validate reference and release reason.
4. Request ledger release posting or approved reversal from Ledger Service.
5. Mark reservation `released` after ledger confirmation.
6. Update projection from ledger event.
7. Publish `wallet.reservation_released`.

### Expire Reservation Workflow

1. Background worker finds expired active or pending reservations.
2. For active ledger-backed reservations, request release posting.
3. Mark reservation `expired` after successful release.
4. Publish event and metric.

### Cancel Reservation Workflow

1. Validate cancellation is allowed by reservation type.
2. Release ledger hold if active.
3. Mark reservation `cancelled`.
4. Publish event.

### Failure Handling

- Ledger posting failure marks reservation `failed`.
- Projection update failure triggers retry and alert.
- Idempotency conflict returns existing reservation result when safe.
- Expired pending reservations are cleaned up by worker.

### Business Rules

- Reservations must be positive amounts.
- Reservations must match wallet-account currency.
- Closed, frozen, suspended, or archived wallets cannot create new reservations.
- Restricted wallets require policy-specific allow rules.
- Compliance holds require elevated authority.
- Reservation release must never create extra spendable money; it must follow ledger rules.

## SECTION 13 — Security

M3 must protect wallets from unauthorized access, balance tampering, race conditions, duplicate requests, privilege escalation, and wallet hijacking.

### Unauthorized Access Controls

- Enforce authentication on every wallet endpoint.
- Enforce owner, business-role, service-account, or admin authorization.
- Deny cross-tenant wallet access.
- Return safe errors that do not reveal wallet existence.

### Balance Tampering Controls

- No direct balance write endpoints.
- Projection updates only from ledger events or authenticated rebuild workers.
- Projection rebuilds compare against Ledger Service.
- Manual corrections require ledger adjustments in M2, not wallet balance edits.

### Race Condition Controls

- Use idempotency keys for create, reserve, release, freeze, unfreeze, close, and archive operations.
- Use database transactions for wallet metadata changes.
- Use optimistic locking or row locks for reservation state transitions.
- Use ledger atomicity for financial holds.

### Duplicate Request Controls

- Store idempotency keys per wallet/operation.
- Return existing result for safe duplicate requests.
- Detect conflicting duplicate payloads.

### Privilege Escalation Controls

- Wallet type creation requires explicit permission.
- Admin actions require RBAC and MFA where inherited from M1 policy.
- System wallet operations require service identity or Super Admin authority.
- Role checks must happen server-side.

### Wallet Hijacking Controls

- Wallet ownership cannot be changed through ordinary update APIs.
- Owner changes require controlled migration/admin process and audit.
- Suspicious access patterns must be logged and alertable.

### Data Protection

- Do not log full balances in high-volume logs unless needed and safe.
- Redact owner identifiers where appropriate.
- Protect wallet metadata through least-privilege database roles.
- Encrypt sensitive configuration.

### Security Events

Emit or record security events for:

- Unauthorized wallet access attempt.
- Wallet freeze/unfreeze.
- Wallet closure.
- Limit override.
- Reservation failure.
- Projection drift.
- Admin wallet search.

## SECTION 17 — Performance

M3 must be designed for millions of wallets and real-time balance lookups.

### Scale Targets

- Millions of wallets.
- Multiple wallet accounts per wallet.
- High-frequency balance reads.
- High-volume projection updates from ledger events.
- Concurrent reservations against active wallets.

### Real-Time Balance Lookups

Requirements:

- Balance endpoint should read from projection table/cache.
- Projection response must include freshness metadata.
- If projection is stale beyond threshold, API may trigger asynchronous rebuild or return stale warning according to policy.

### Fast Balance Projection

Requirements:

- Process ledger events idempotently.
- Update only affected wallet accounts.
- Store last processed ledger journal/version.
- Batch projection updates where safe.

### Read Replicas

- Use read replicas for wallet list, search, and analytics-style reads when scale requires.
- Balance reads may use primary or strongly consistent read path depending on freshness requirements.

### Caching

Cache:

- Wallet summaries.
- Balance projections.
- Currency definitions.
- Limit policies.

Rules:

- Cache invalidation must happen on wallet event and ledger projection update.
- Cache TTLs must be short for balances.
- Cache must not become source of truth.

### Snapshot Optimization

- Use wallet balance projections as snapshots for fast reads.
- Historical snapshots may be moved to analytics warehouse.
- Full rebuild must be possible from ledger.

### Horizontal Scaling

- API handlers should be stateless.
- Projection workers should partition by ledger account, wallet ID, or event partition.
- Reservation workers must coordinate idempotency and locking.
- Event consumers must support replay.

### Performance Targets

- `GET /wallets/{id}/balance` p95 under 150ms from projection cache/table under normal load.
- `GET /wallets` p95 under 250ms for paginated normal user lists.
- `POST /wallets` p95 under 500ms excluding Ledger Service account creation latency.
- Reservation creation p95 under 500ms excluding Ledger Service posting latency.
